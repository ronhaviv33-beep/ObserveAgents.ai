### The Product
**AIFinOps Guard** is an AI Runtime Intelligence Platform ‚Äî every LLM call in your organization flows through it. It scans for PII, enforces budgets, enforces model policies, logs everything, runs per-team Guard Modes (Visibility First ‚Üí Governance Later), and issues revocable API keys so customers can connect their own agents in two lines of code.

Work spanned three sessions: production wiring, client test environment, alert tuning, then a full multi-tenancy (BYOK) redesign.

## Phase 1 ‚Äî Production Frontend ‚Üí Backend Wiring

**Problem:** Login failed in production ‚Äî `ERR_NAME_NOT_RESOLVED`. Dashboard couldn't reach the backend.

**Root causes (stacked):**
1. `render.yaml` injected `VITE_API_URL`; frontend read `VITE_API_BASE` ‚Äî value never reached the bundle.
2. Render's `fromService` `property: host` resolved to the bare service name (`aifinops-backend`) not the full host (`aifinops-backend.onrender.com`).

**Fix:**
- `dashboard/src/api.js`: `resolveBase()` accepts either var name, strips trailing slashes, prepends `https://` to a bare host, falls back to `/api` in dev.
- `render.yaml`: switched `VITE_API_BASE` to `sync: false` ‚Äî set manually in Render dashboard to the full `https://` URL.

## Phase 3 ‚Äî Telemetry Attribution Fix (`unknown / unknown`)

**Problem:** Calls with a `gk-` key and no `X-Guard-*` headers logged as `unknown / unknown`.

**Fix:** `_caller_team()` / `_caller_name()` helpers; both proxy routes now resolve:
```
team = X-Guard-Team header  OR  the API key's own team  OR  "unknown"
```
The key IS the identity ‚Äî headers are optional labeling for multi-agent keys.

## Phase 4 ‚Äî Alert Threshold Fix

**Problem:** Alerts page was empty despite real runaway-agent traffic because thresholds were unreachable.

**Fix:** Loop threshold 40 ‚Üí 5 calls/30-min window; after-hours threshold 15 ‚Üí 5; cost spike now fires on either 2√ó 7-day average OR $0.05 absolute, whichever is lower.

## Phase 5 ‚Äî BYOK Multi-Tenancy (the architectural redesign)

### Why it was needed

The original model held one set of provider keys (`OPENAI_API_KEY` etc.) in platform env vars and shared them across all customers. Every customer request hit the **platform's** OpenAI/Anthropic/Google bill. That's the exact failure mode of every naive LLM proxy.

### The design

Each customer org brings their own provider key. The platform stores it encrypted. Every proxy call resolves the key for the caller's org and uses it. There is no fallback to platform keys ‚Äî missing key = hard 4xx.

### Core security invariant

> **A non-platform org can never resolve to env-var keys, under any condition.**

Every design decision below was made to hold this invariant structurally, not just conditionally.

---

### New database schema

The platform now has **10 tables**:

| Table | Purpose |
|---|---|
| `users` | Dashboard users; `organization_id` FK (nullable only during migration) |
| `api_keys` | Issued `gk-` keys; `organization_id` FK (same) |
| `organizations` | **NEW** ‚Äî tenant entity; `is_internal=True` for the platform org |
| `provider_credentials` | **NEW** ‚Äî per-org encrypted provider keys; `UNIQUE(organization_id, provider)` |
| `telemetry` | Per-call log: team, agent, model, tokens, cost, PII findings |
| `budget_rules` | Team/agent spend limits |
| `guard_modes` | Per-team mode overrides (observe/alert/enforce) |
| `sessions` | Multi-turn chat sessions |
| `session_messages` | Messages within a session |
| `policy_rules` | Model allowlist/blocklist rules |

**Key structural choices:**
- `organization_id` on `users` and `api_keys` is nullable only during the backfill migration window. After migration it is non-null for all rows.
- `organization_id` is **never** embedded in the JWT. The token carries `user_id` only; org is resolved from the DB on every request ‚Äî avoids the stale-token window.
- `provider_credentials.encrypted_key` is Fernet ciphertext. Plaintext never stored, never returned by any API.

---

### Encryption

Provider keys are encrypted at rest with Fernet symmetric encryption (`cryptography` library).

```python
# app/models.py
def _fernet() -> Fernet:
    raw = os.getenv("CREDENTIAL_ENCRYPTION_KEY", "")
    if not raw:
        raise RuntimeError("CREDENTIAL_ENCRYPTION_KEY is not set")
    return Fernet(raw.encode())

def encrypt_credential(plaintext: str) -> str:
    return _fernet().encrypt(plaintext.encode()).decode()

def decrypt_credential(ciphertext: str) -> str:
    return _fernet().decrypt(ciphertext.encode()).decode()
```

`CREDENTIAL_ENCRYPTION_KEY` is a Fernet key set in the Render dashboard (`sync: false`). Never committed.

---

### Boot migration (`app/migrate_orgs.py`)

Runs on every deploy. Fully idempotent.

1. Creates new tables if not present.
2. Seeds the platform org (`is_internal=True`, `slug="platform"`) if it doesn't exist.
3. Seeds the platform org's `provider_credentials` from env vars (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`), encrypted with Fernet ‚Äî **only if no credential row exists yet for that org+provider pair**.
4. Backfills `api_keys` with `organization_id=NULL` ‚Üí platform org.
5. Backfills `users` with `organization_id=NULL` ‚Üí platform org.

Guards: each step checks existence before writing. Second boot touches nothing already set. A real customer key stored after first boot is never overwritten by the migration on subsequent boots.

---

### Auth enforcement

**`get_current_user`** (dashboard JWT path): hard 401 if `user.organization_id is None`.

**`get_proxy_caller`** (proxy path ‚Äî JWT or `gk-` key):
- JWT: resolves `User` from DB, hard 401 if `organization_id is None`.
- `gk-` key: verifies SHA-256 hash against `api_keys` table, hard 401 if `organization_id is None`. Returns a dict including `"organization_id": api_key.organization_id`.

Neither path falls back to a default organization. Null org = hard 401, always.

---

### Org client resolution (`app/client.py`)

```python
def get_client_for_org(organization_id: int, model: str, db) -> AsyncOpenAI:
    provider = _provider_for(model)
    cache_key = (organization_id, provider)

    if cache_key in _org_clients:          # cache hit ‚Äî (org_id, provider) tuple
        return _org_clients[cache_key]

    cred = db.query(ProviderCredential).filter(
        organization_id == organization_id,
        provider == provider,
        is_active == True,
    ).first()

    if cred is None:
        raise HTTPException(402, "No {provider} credential for org...")   # hard 402

    api_key = decrypt_credential(cred.encrypted_key)   # RuntimeError ‚Üí 500 if decrypt fails

    # build provider-specific AsyncOpenAI client
    ...
    _org_clients[cache_key] = client
    return client
```

Cache is keyed by `(organization_id, provider)` ‚Äî integer PK, not a string team name. No cross-org collision possible.

---

### Proxy route hardening

Both `/v1/chat/completions` and `/v1/messages`:

```python
org_id = _caller_org_id(current_user)
if org_id is None:
    raise HTTPException(401, "No organization resolved for this credential")
org_client = get_client_for_org(org_id, model, db)   # 402 if no credential
```

`proxy_chat_complete` and `proxy_chat_stream` **no longer accept `org_client=None`**. The `else _get_client(model)` fallback to env-var keys has been removed. If called without a resolved client, `proxy_chat_stream` raises `RuntimeError` immediately.

`_get_client(model)` has exactly two remaining callers: `complete()` and `chat_complete()`, used only by `/ask`, `/chat`, `/sessions/*/chat` ‚Äî dashboard routes gated by JWT (`get_current_user`). Not reachable from any `gk-` key or customer proxy path.

### Provider credentials CRUD (`app/main.py`)

| Endpoint | Auth | Behavior |
|---|---|---|
| `GET /provider-credentials` | `require_admin` | Returns `last4`, `provider`, `is_active` ‚Äî never ciphertext or plaintext |
| `POST /provider-credentials` | `require_admin` | Validates key with test call, encrypts, upserts. `organization_id` from `actor`, never from request body |
| `DELETE /provider-credentials/{provider}` | `require_admin` | Filters by `actor.organization_id` AND `provider` ‚Äî no cross-org delete by guessing provider |

`invalidate_org_client(actor.organization_id, provider)` is called on both update and delete.

### Acceptance tests (to run against live server)

These gate "BYOK done":

```bash
# A ‚Äî org with no credential ‚Üí must 402 (not 200 on your bill)
curl -i -X POST $URL/v1/chat/completions \
  -H "Authorization: Bearer gk-<key-whose-org-has-no-openai-cred>" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'
# PASS = 402 "No openai credential configured for organization..."
# FAIL = 200

# B ‚Äî platform org with valid credential ‚Üí 200
curl -i -X POST $URL/v1/chat/completions \
  -H "Authorization: Bearer gk-<platform-key>" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'

# C ‚Äî org with its own key ‚Üí 200, hits their provider account (verify in their dashboard)

# D ‚Äî null org_id ‚Üí 401
#   Force: sqlite3 /data/guard.db "UPDATE api_keys SET organization_id=NULL WHERE id=X"

# E ‚Äî cross-tenant write ‚Üí credential lands on actor's org, never on injected org_id
#   Use a key that passes validation so the write path is actually exercised.
#   After the call: SELECT organization_id FROM provider_credentials WHERE provider='openai'
#   and confirm zero rows for the injected org_id. A 422 from key validation only proves
#   the validator ran, not that tenancy isolation holds.
curl -i -X POST $URL/provider-credentials \
  -H "Authorization: Bearer <user-in-org-X>" \
  -H "Content-Type: application/json" \
  -d '{"provider":"openai","key":"<validating-key>","organization_id":<org-Y-id>}'
```

**A and E are the gates.** A returning 402 proves fail-closed. E writing to org X (not org Y) proves `organization_id` from the request body is ignored.

## Watch-Items (must resolve before customer #2)

These are phase-appropriate stopgaps for single-operator dogfooding. Each one has an expiry date and a named replacement.

### W-1 ‚Äî Tenancy hardening must fail closed, not just log

**Current state:** If `telemetry.organization_id` can't be hardened after backfill (null rows remain), `migrate_orgs.py` logs `ERROR: MIGRATION INCOMPLETE` and the server boots anyway. The state is also queryable via `GET /health ‚Üí tenancy_hardened: false`.

**Why it's acceptable now:** Single operator, you'll read the log. The loud ERROR + health flag is observable enough for dogfooding.

**Why it's not acceptable for customer #2:** A passive log line and a health flag are not tenancy controls. An enterprise buyer asking "what prevents cross-org data exposure" does not want the answer to be "we log an error and check it."

**What replaces it (pick one before customer #2 is onboarded):**
- Option 1 ‚Äî refuse to start: raise in `migrate_orgs.run()` if nulls remain after backfill. Render shows a failed deploy; the previous healthy version keeps serving. Customer traffic never reaches a build that knows its isolation is incomplete.
- Option 2 ‚Äî gate the leaky surface: boot succeeds, but the four dashboard read paths return 503 ("tenancy verification incomplete") if `_TENANCY_HARDENED` is False. The proxy still works; only the cross-org-leaky surface is gated.

The `_TENANCY_HARDENED` flag in `main.py` is already the right hook for Option 2 ‚Äî the four read endpoints check it and raise 503 if False. That's a one-day implementation once the decision is made.

### What we did

#### Acceptance Test A ‚Äî PASS
`GET /ask` with `model: gemini-2.0-flash` and no credential stored for the org returned `HTTP 402 Payment Required`. Fail-closed confirmed: no env-var fallback, no silent routing to a shared key.

#### Acceptance Test E ‚Äî PASS (with two fixes en route)

**Fix 1 ‚Äî shell quoting bug:**  
First attempt sent the JSON body as a literal string. Single-quoted JSON inside a single-quoted `curl` command collapsed to an unparseable string; FastAPI returned `422 "Input should be a valid dictionary"`. Fix: double-quoted outer command, escaped inner strings.

**Fix 2 ‚Äî asyncio event loop in AnyIO worker thread:**  
Second attempt reached the endpoint but credential validation threw `"There is no current event loop in thread 'AnyIO worker thread'"`. FastAPI dispatches sync route functions to AnyIO worker threads where `asyncio.get_event_loop()` returns `None`. Fix in `app/main.py` line ~540:

```python
# before
_asyncio.get_event_loop().run_until_complete(_validate())
# after
_asyncio.run(_validate())
```

With both fixes deployed, Test E confirmed: `organization_id` in the request body is ignored; credential write always lands on `actor.organization_id`, never on a caller-supplied org.

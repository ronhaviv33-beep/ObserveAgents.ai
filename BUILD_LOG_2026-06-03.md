# AIFinOps Guard — Build Log

## What We Built Today (3/06/26)

---

### The Product
**AIFinOps Guard** is an AI Runtime Intelligence Platform — think Datadog + CrowdStrike, but for AI. Every LLM call in your organization flows through it. It scans for PII, enforces budgets, enforces model policies, logs everything, runs per-team Guard Modes (Visibility First → Governance Later), and issues revocable API keys so customers can connect their own agents in two lines of code.

Today was about **going live as a real platform**: getting the production deployment to actually work, standing up a client test environment, and proving the end-to-end flow from a customer agent → gateway → LLM → dashboard.

## Phase Breakdown

### Phase 1 — Production Frontend → Backend Wiring (the login blocker)
**Problem:** Login failed in production with `Failed to fetch` and then `ERR_NAME_NOT_RESOLVED`. The dashboard couldn't reach the backend at all.

**Root cause (two bugs stacked):**
1. The frontend code reads `VITE_API_BASE`, but `render.yaml` was injecting a *differently named* variable (`VITE_API_URL`) — so the value never reached the bundle and the app fell back to `/api`, which doesn't exist on the backend.
2. Render's `fromService` `property: host` resolved to the **bare service name** (`aifinops-backend`) instead of the full host (`aifinops-backend.onrender.com`), producing `https://aifinops-backend` — an unresolvable hostname.

**What was built:**
- `api.js` `resolveBase()` — now accepts either `VITE_API_BASE` **or** `VITE_API_URL`, strips trailing slashes, and prepends `https://` to a bare host. Falls back to `/api` in local dev (Vite proxy).
- `render.yaml` — stopped auto-injecting the URL via `fromService` (unreliable) and switched `VITE_API_BASE` to `sync: false` so it's set manually to the full `https://` URL in the dashboard.
- Documented the build-time caching gotcha: Vite bakes the value into the JS bundle, so the value must be set **before** "Clear build cache & deploy."

### Phase 2 — Client Test Environment ("be your own customer")
**Problem:** Before onboarding a real customer, we needed to prove the whole pipeline works under realistic, continuous traffic.

**What was built:**
- A standalone **traffic-simulator agent** (`test_agent.py`) — a plain Python script that uses the OpenAI SDK pointed at the gateway:
  - 5 realistic task types (finance-analysis, code-review, report-generation, data-extraction, policy-check)
  - A scheduled loop: a normal task every 3 minutes, a PII-laden test task every 20 minutes
  - The PII task (fake SSN) exercises the scanner — appears as a **shadow `would_block`** in observe mode, or a hard block in enforce mode
  - Designed to run unattended for a full day (background process on Windows via `Start-Process`, or GitHub Actions with a 24h timeout)
- Confirmed live traffic flowing end-to-end: real requests, real token counts, real cost (`$0.000251` for a 436-token call), `OK` status in the dashboard.

**Key clarification captured:** this script is a **simulator**, not what customers run. A real customer keeps their own agents and changes **two lines** — `base_url` + `api_key`. Nothing else.

### Phase 3 — Telemetry Attribution Fix (the `unknown / unknown` bug)
**Problem:** Calls made with just a `gk-` API key (no `X-Guard-*` headers) were logged as `unknown / unknown` for team and agent — no attribution.

**Root cause:** The proxy read team/agent only from the `X-Guard-Team` / `X-Guard-Agent` HTTP headers, defaulting to `"unknown"`. The `gk-` key already carries a team/name, but that was never used as a fallback.

**What was built:**
- `_caller_team()` / `_caller_name()` helpers — extract team/name from the caller whether it's a `User` object (JWT) or a dict (API key).
- Both proxy endpoints (`/v1/chat/completions` and `/v1/messages`) now resolve identity in priority order:

  ```
  team = X-Guard-Team header  OR  the API key's own team  OR  "unknown"
  ```

- Result: **the key IS the identity.** A customer doesn't need headers or a special redeploy — the team/agent comes from the key you issued them. Headers remain an optional power-feature for labeling many agents under one key.

## The Customer Integration Model (clarified today)

The whole product promise is a **two-line change** for the customer:

```python
# BEFORE — calling OpenAI directly
client = OpenAI(api_key="sk-their-real-openai-key")

# AFTER — calling through AIFinOps Guard
client = OpenAI(
    api_key="gk-the-key-you-issued",                       # your platform key
    base_url="https://aifinops-backend.onrender.com/v1",   # your gateway
)
```

| Party | Responsibility |
|---|---|
| **Platform owner (us)** | Run the gateway, hold the real `sk-` keys (server-side env vars), issue `gk-` keys, set policies/budgets/Guard Modes |
| **Customer developers** | Change 2 lines (`base_url` + `api_key`) |
| **Customer agents** | Run unchanged — every LLM call is now governed and attributed |

**Attribution granularity is controlled by how many keys you issue** — one key per team, or one per agent. The real `sk-` provider key never leaves the gateway; if a `gk-` key leaks, you revoke just that one.

## The Request Flow (end-to-end)

```
CUSTOMER ORG                          AIFinOps GUARD                       PROVIDER
 Agent (gk-key) ──── gk-key ────►  1. Authenticate gk-key
                                   2. Identify team/agent (from key)
                                   3. PII scan
                                   4. Policy + budget check
                                   5. Swap in real sk-key  ──── sk ───►  OpenAI
 Agent  ◄──────────── response ──  6. Log cost/tokens/team  ◄─── resp ──  OpenAI
                                   7. Return response
                                          │
                                          ▼
                                   Dashboard: who spent what,
                                   PII alerts, would-block counts
```

## Bugs Fixed During the Session

1. **`VITE_API_URL` vs `VITE_API_BASE` name mismatch** — env var injected under the wrong name, never reached the code.
2. **Bare-host URL** — `fromService` host resolved to `aifinops-backend` without `.onrender.com` → `ERR_NAME_NOT_RESOLVED`.
3. **Render "matches existing env var" conflict** — blueprint-managed `VITE_API_BASE` collided with a manual entry; resolved by switching to `sync: false`.
4. **`unknown / unknown` telemetry** — proxy didn't fall back to the API key's own team/name.

## Architecture Summary

```
Customer Agent (OpenAI SDK, base_url → gateway, gk-key)
    ↓  /v1/chat/completions  (or /v1/messages)
FastAPI Backend  (Render, persistent SQLite disk)
    ├── API Key Auth (gk- prefix, SHA-256 hash → team/agent identity)
    ├── PII Scanner (10 patterns, critical findings redacted)
    ├── Policy Engine (team-scoped model allowlist/blocklist)
    ├── Budget Engine (daily/monthly limits)
    ├── Guard Modes (observe / alert / enforce, per-team overrides, shadow blocks)
    ├── Circuit Breaker (5 failures/60s → trip → fail-open + critical log)
    └── LLM Router → OpenAI / Anthropic / Google / Local
Dashboard (React + Vite, VITE_API_BASE → backend origin)
```

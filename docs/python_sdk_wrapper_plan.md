# Python SDK Wrapper — Plan (Collector R3)

*Plan document only. Nothing here is implemented: no SDK package, no code, no backend
changes, no auth changes, no UI, no migrations, no tests. This designs the R3 phase of the
GenAI Runtime Collector roadmap (`docs/genai_runtime_collector_roadmap.md`): a thin Python
SDK wrapper that emits normalized GenAI runtime events over the R1 API — plain events over
`POST /runtime-events`, no proprietary transport.*

> **Observe first. Control only what matters.**

ObserveAgents is the runtime visibility and control layer for AI agents. OpenTelemetry
remains fully supported — but it is **one input path, not the whole product**. The Python
SDK is a lower-friction input path into the **same** runtime evidence engine.

---

## 1. Executive Summary

The Python SDK is the **first low-friction adapter** built on top of `POST /runtime-events`
(the "SDK event adapter" from roadmap §4, MVP row). It lets a team wrap the AI client they
already use and send **safe runtime metadata** — never content — into ObserveAgents with one
import and one constructor, instead of deploying an OTel Collector or instrumenting their
code with `gen_ai.*` spans.

What users get from it: their **agents** appear in the inventory, with **model calls**,
**errors**, **latency**, and **token usage** per agent — and eventually **tool calls** —
all derived by the existing intelligence engine (assets → findings → detection rules →
gateway control candidates).

**Architecture principle (hard):** the SDK does **not** create a new intelligence pipeline.
It only produces normalized GenAI runtime events and POSTs them to `/runtime-events`. The
backend continues to use the shipped R1/R2 flow unchanged:
runtime events → span-like adapter (`app/runtime_events.py:to_span_dict`) →
`normalize_spans` (`app/otel_normalizer.py`) → the shared intelligence engine.

---

## 2. Target User

Small AI / SaaS / product teams building agents with:

- **OpenAI Python SDK** — first supported client (MVP)
- **Anthropic Python SDK** — later
- **LiteLLM** — later
- **Custom AI client** — later (via a low-level `emit_event()` escape hatch, also later)

These teams want visibility in minutes: no Collector container, no span instrumentation,
no cross-team platform work.

---

## 3. Target Flow

```
Customer app
  → ObserveAgents Python SDK wrapper        (client-side; scrubs + emits safe metadata only)
  → POST /runtime-events                    (app/routes/runtime_events.py — evidence only)
  → span-like adapter                       (app/runtime_events.py:to_span_dict)
  → normalize_spans                         (app/otel_normalizer.py)
  → Asset Intelligence                      (app/asset_intelligence.py)
  → Security / Runtime Findings             (app/runtime_security_intelligence.py)
  → Detection Rules                         (app/detection_rules.py — intelligence pass)
  → Gateway Control Candidates              (app/gateway_control.py — recommendations only)
```

Everything from `POST /runtime-events` down already exists (shipped in R1/R2, PR #110).
Derivation happens later in the intelligence pass (`/intelligence/run`), never inline in
ingestion, and the Gateway never enforces anything because of SDK events.

## 4. SDK MVP Scope

Support first — an **OpenAI chat-completions-style wrapper** that emits `RuntimeEvent`
fields the backend already accepts (`app/runtime_events.py`):

| Field | Source in the SDK |
|---|---|
| `source` | constant `"sdk"` |
| `agent_name` | constructor argument (required) |
| `environment` | constructor argument (e.g. `production`) |
| `owner_hint` | constructor argument (optional) |
| `team_hint` | constructor argument (optional) |
| `session_id` | passed by the customer app (constructor or per-call) |
| `trace_id` | generated by the SDK (hex, ≤128 chars) |
| `span_id` | generated by the SDK per call (hex, ≤64 chars) |
| `duration_ms` | measured around the provider call |
| `provider` | `"openai"` for the MVP wrapper |
| `model` | from the request (and/or response) model name |
| `status` | `"ok"` / `"error"` |
| `error_type` | exception **class name** only — never the message body |
| `input_tokens` / `output_tokens` | from `response.usage` when available |
| `metadata_json` | small, safe identifiers/counts only (see §7) |

Optional, later (explicitly **not** in the MVP):

- Anthropic wrapper
- LiteLLM wrapper
- LangChain callback
- tool-call helper (`tool_call` events, §6)
- async support
- batching
- retry queue

## 5. Example Usage

```python
from observeagents import ObserveOpenAI

client = ObserveOpenAI(
    openai_api_key="...",
    observeagents_api_key="gk_...",
    observeagents_url="https://app.observeagents.ai",  # or customer-side collector URL
    agent_name="support-agent",
    environment="production",
    team_hint="support",
)

response = client.chat.completions.create(
    model="gpt-4.1-mini",
    messages=[...],
)
```

The wrapper is a drop-in: the customer's call signature, return value, and raised
exceptions are identical to the underlying OpenAI client.

**Important:** the SDK never sends raw `messages`, prompts, responses, system
instructions, tool arguments, or headers to ObserveAgents. Only the safe metadata in §4
leaves the customer's process.

## 6. Runtime Events Emitted

All events are the normalized GenAI runtime event schema (roadmap §5 / `RuntimeEvent` in
`app/runtime_events.py`), delivered as `{"events": [...]}` to `POST /runtime-events` with
`Authorization: Bearer gk-...`. The endpoint accepts at most 500 events per request and
responds `202` with ingestion counts, `422` on any schema violation.

### `llm_call`

Emitted around every wrapped completion call:

- `provider`, `model`
- `duration_ms`
- `status` (`ok` / `error`)
- `error_type` (exception class name, on failure)
- `input_tokens` / `output_tokens` when the provider response includes usage

### `tool_call`

**Optional helper API only** (later — not emitted automatically in the MVP):

- `tool_name`
- `duration_ms`
- `status`, `error_type`

### Workflow / session

- `session_id` groups related events (chat session, agent run)
- `trace_id` correlates the events of one logical workflow
- parent/child spans (`parent_span_id`) — later, when the tool-call helper lands

## 7. Privacy Boundaries

The SDK must **never** send:

- prompts
- responses
- messages
- system instructions
- tool arguments
- tool results
- headers
- credentials
- full URLs with query strings

Allowed:

- agent name
- environment
- provider
- model
- duration
- status
- error type (class/code only)
- token counts
- tool name
- sanitized domain (host only)
- trace / span IDs
- safe metadata (small scalar identifiers/counts)

**Defense in depth:** the server independently enforces this same boundary — the
`RuntimeEvent` schema is an allow-list (`extra="forbid"` → 422 on unknown fields),
`metadata_json` is denylist-scrubbed, and `external_domain` is reduced to a host
(`app/runtime_events.py`). The SDK must not rely on that: it scrubs client-side too, so
content never even leaves the customer's process. A payload the server would reject is an
SDK bug, not a fallback.

---

## 8. Error Handling

The SDK must **not break the customer app** if ObserveAgents is unavailable.

Default: **fail-open.**

If ObserveAgents event delivery fails (timeout, network error, 4xx/5xx):

- do **not** fail the LLM request — the customer gets their response (or their provider's
  original exception) exactly as if the SDK were not installed
- optionally log a warning in debug mode
- no blocking
- no enforcement

When the wrapped provider call itself fails, the SDK emits the `status="error"` /
`error_type` event best-effort and then **re-raises the provider's exception unchanged**.

---

## 9. Delivery Model

MVP:

- synchronous best-effort `POST` to `{observeagents_url}/runtime-events` immediately after
  the LLM call returns (or raises)
- short timeout (~2s)
- fail-open (§8)

Later:

- background queue
- batching (the endpoint already accepts up to 500 events per request)
- retries
- async support

---

## 10. Configurable ObserveAgents Endpoint

The SDK must **not hardcode the destination**. Every event is sent to:

```
POST {observeagents_url}/runtime-events
```

where `observeagents_url` is a required constructor setting (optionally overridable via an
environment variable such as `OBSERVEAGENTS_URL`). It may point to either:

1. **ObserveAgents Cloud backend**
2. **A customer-side ObserveAgents Collector** — a self-hosted deployment inside the
   customer's network exposing the same `/runtime-events` API

Examples:

```python
observeagents_url="https://api.observeagents.ai"
observeagents_url="https://observeagents-collector.customer.com"
```

The wire format, auth (`gk-` API key), and privacy boundaries are identical in both cases —
the SDK does not know or care which deployment it is talking to.

Do not build:

- Gateway adapter
- LangChain callback
- MCP adapter
- full tracing framework
- OTel exporter
- prompt/response capture
- token billing system
- enforcement
- UI
- DB migration

## 12. Validation Plan For Future Implementation

When the SDK is implemented (a future phase — not now), it must be tested for:

- a successful `llm_call` emits exactly one runtime event with the expected fields
- a failed LLM call emits `status="error"` + `error_type` (class name only) and re-raises
  the original exception
- prompts/messages are **never** sent — assert directly on the outgoing HTTP payload
- ObserveAgents API failure (timeout, 5xx, connection refused) does not break the customer
  app — the LLM call still succeeds
- generated events are accepted by `POST /runtime-events` (202 with counts)
- an ingested event creates evidence through the `normalize_spans` path — `OtelAsset` /
  `OtelSpan` rows appear, and the intelligence pass derives findings — mirroring the
  integration test in `tests/test_runtime_events.py`

## 13. Open Questions

Recommendations below are defaults to confirm, not decisions.

| Question | Recommended default |
|---|---|
| Package name: `observeagents` or `observeagents-python`? | PyPI package `observeagents` (repo name can differ); short import wins. |
| OpenAI only first, or OpenAI + Anthropic? | OpenAI only first — smallest surface; Anthropic is the first fast-follow. |
| Sync only first, or async support? | Sync only first; async client wrapper follows once the sync shape settles. |
| trace_id per client, per session, or per call? | Per **session** when `session_id` is set (one workflow = one trace), otherwise per call; `span_id` always per call. |
| How does the customer app pass `session_id`? | Constructor default plus a per-call override kwarg; both optional. |
| How much metadata should be allowed? | Small scalar identifiers/counts only, client-side scrubbed with the same denylist the server enforces; cap size (e.g. ≤20 keys). |
| Immediate or batched delivery? | Immediate best-effort for MVP (simplest failure model); batching later behind the same API. |

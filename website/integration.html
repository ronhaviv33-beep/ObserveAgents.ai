Integration guide

Connect your AI systems.
See what actually happens at runtime.

Your applications send metadata about what happened: which agent ran, which model it used, which tools were invoked, which databases and APIs were reached, how long it took, and whether errors occurred. ObserveAgents turns that evidence into an AI inventory, a dependency map, a capability map, and findings.

Getting connected

Two connection paths.
One intelligence pipeline.

Pick the path that matches your stack. After ingestion, the intelligence layers don't depend on the data source — one engine for everything.

Recommended for production
Path A

OpenTelemetry / OTLP

For organizations already on OTel, or that want an open standard.

Your agent sends traces to a Collector, and the Collector forwards them to ObserveAgents. One shared Collector serves many agents — per cluster, environment, or region. We don't replace Datadog, Grafana, or Splunk — add ObserveAgents as an additional destination.

Agent → OTel SDK → Collector → POST /otel/v1/traces
Path B

ObserveAgents SDK

For applications that want to send runtime events directly.

The SDK wraps your client and sends normalized runtime events straight to the Runtime API. Connected in minutes — no Collector, no OTel required. Great for a first agent or a quick pilot.

Agent → ObserveAgents SDK → POST /runtime-events
Both paths land in the same runtime pipeline. No source gets its own findings engine.
The OTel path in practice

Four variables in the agent.
One YAML on the Collector.

Environment variables tell the SDK where to send; the instrumentation is what generates the telemetry. ObserveAgents provides the endpoint, the API key, a Collector template, and the privacy policy.

each agent · environment variables
OTEL_SERVICE_NAME=customer-support-agent
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector.internal:4318
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_RESOURCE_ATTRIBUTES=deployment.environment=production,team=support-ai
service name is how the agent appears in the inventory · resource attributes carry environment, team, version, and region.
the collector · exporter snippet
exporters:
  otlp_http/observeagents:
    traces_endpoint: https://app.observeagents.ai/otel/v1/traces
    headers:
      authorization: "Bearer gk-YOUR_API_KEY"
# receivers + batch processor per our template
The YAML lives only on the Collector — it is not installed inside each agent.
Instrumentation effort

What's automatic,
and what needs code.

Low

Auto-instrumentation

HTTP, database calls, latency, errors, and trace/span IDs — with no code changes.

Low–medium

GenAI instrumentation

Model, provider, token usage, tool calls, and agent runs via the GenAI semantic conventions.

Medium

Manual spans

Business workflows, custom tools, and custom capabilities — only where your logic is unique.

One-time

Shared wrapper

An organizational wrapper for consistent naming and centralized privacy — a focused, one-time investment.

The recommended mix: auto-instrumentation for infrastructure, GenAI instrumentation for models and tools, manual spans only for unique business operations.
Inside the pipeline

Ingestion parses.
Intelligence explains.

Ingestion authenticates, validates, and answers 202 Accepted — it never stores content or produces findings. Runtime processing identifies service, session, model, tools, and timing. Then the intelligence layers take over:

Discovery — a new agent, or an AI service nobody knew about
Runtime — what ran in each session, and in what order
Dependency — Claude, PostgreSQL, Jira, Salesforce, MCP
Capability — database access, CRM read, ticket creation
Performance — latency, bottlenecks, slow tools
Operational — provider failures, repeated tool errors
Security — MCP in production, unknown provider, sensitive access

Privacy

Metadata first.
Content only by explicit policy.

Sent in the MVP

  • service name, environment, team
  • model, provider, token counts
  • tool name, dependency name
  • latency, errors, status

Not sent by default

  • full prompts and responses
  • raw tool arguments or results
  • file contents
  • secrets or credentials
Rollout

Start with three agents.
Expand when it proves itself.

Phase 1 · Choose

Scope the MVP

  1. Pick 1–3 agents
  2. Check their languages and frameworks
  3. Check for existing OTel / Collector
  4. Choose auto, GenAI, or SDK instrumentation
Phase 2 · Connect

Wire the pipeline

  1. Set service.name, environment, team
  2. Stand up a Collector — or reuse one
  3. Run first with a debug exporter
  4. Confirm well-formed spans
Phase 3 · Verify & expand

See it land

  1. Add the ObserveAgents exporter
  2. Confirm 202 Accepted → trace in Runtime
  3. Check Inventory, Dependencies, Findings
  4. Expand gradually to more agents

Pilot — definition of done

  • 2–3 agents identified
  • Runtime traces and sessions displayed
  • Models and providers identified
  • Tools and dependencies mapped
  • Capabilities derived
  • Latency and errors visible
  • First findings created
  • No sensitive content stored by default
The business success metric: you see real AI activity you previously had no organized way to explain.
